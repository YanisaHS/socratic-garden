# Installing Socratic Garden across tools

Socratic Garden has two layers, and they travel differently between tools:

- **Skills** — the reusable disciplines under `.github/skills/`. They follow the
  [Agent Skills](https://agentskills.io/) standard, so any skills-compatible tool
  can load them.
- **Agent modes** — the six workflows under `.github/agents/`. They use the
  VS Code / GitHub Copilot custom-agent convention. Tools without custom-agent
  support don't read them; there, you use the skills directly or the fallback CLI.

Pick the option below that matches your tool. The project config,
`socratic-garden.yaml`, is optional everywhere: the modes use it when it's
available and ask you for the same context when it isn't.

## What goes where

| Layer | Lives in | Read by | Portable |
| --- | --- | --- | --- |
| Skills | `.github/skills/<name>/SKILL.md` | Any skills-compatible tool | Yes — Agent Skills standard |
| Agent modes | `.github/agents/*.agent.md` | VS Code, GitHub Copilot CLI | No — Copilot/VS Code convention |
| Fallback session | generated by the Python CLI | Any chat tool (pasted in) | Yes — manual paste |

## Option A — Portable skills with the skills CLI

For tools other than Copilot, the quickest way to install the skill layer is the
[skills CLI](https://www.skills.sh/):

```bash
# Preview what would be installed.
npx skills add YanisaHS/socratic-garden --list

# Install the skills into the tool(s) it detects.
npx skills add YanisaHS/socratic-garden

# Install one skill, or install at the user level for every project.
npx skills add YanisaHS/socratic-garden --skill grilling
npx skills add YanisaHS/socratic-garden -g
```

Notes:

- The skills CLI is a distribution convenience, **not** a runtime dependency.
  Socratic Garden never calls it while a mode runs.
- It needs Node.js 22.20 or newer.
- The skills live under `.github/skills/`, which isn't one of the skills CLI's
  standard source folders, so it finds them through its recursive search. That
  works, but selective, first-class discovery would be cleaner with a top-level
  `skills/` layout — a change still under consideration, not yet made. Verify
  discovery on a Node 22+ machine before relying on `--skill` selection.

## Option B — VS Code and GitHub Copilot (agent modes + skills)

This is the native experience, and the only one that includes the six agent
modes.

- **Project level:** open this repository in VS Code or run `copilot` from a
  checkout. The files under `.github/agents/` and `.github/skills/` are read
  directly.
- **Personal level (every project you open):** clone once and symlink the modes
  and skills into your personal Copilot directories:

  ```bash
  git clone git@github.com:YanisaHS/socratic-garden.git ~/socratic-garden
  mkdir -p ~/.copilot/agents ~/.copilot/skills
  ln -s ~/socratic-garden/.github/agents/*.agent.md ~/.copilot/agents/
  ln -s ~/socratic-garden/.github/skills/*          ~/.copilot/skills/
  ```

  Because these are symlinks, `git pull` in `~/socratic-garden` updates the modes
  and skills everywhere. A brand-new mode or skill needs one more `ln -s`.

## Option C — Manual per-host skills (no npx)

If you can't run the skills CLI, copy or symlink the skill folders into your
tool's skills directory. Common locations:

| Tool | Project directory | User directory |
| --- | --- | --- |
| Claude Code | `.claude/skills/` | `~/.claude/skills/` |
| Codex | `.agents/skills/` | `~/.codex/skills/` |
| Cursor | `.agents/skills/` | `~/.cursor/skills/` |
| Zed | `.agents/skills/` | `~/.agents/skills/` |
| GitHub Copilot | `.github/skills/`, `.agents/skills/` | `~/.copilot/skills/` |

For example, to symlink every skill into a project's `.claude/skills/`:

```bash
mkdir -p .claude/skills
ln -s "$PWD"/../socratic-garden/.github/skills/* .claude/skills/
```

Use a copy (`cp -r`) instead of a symlink if your tool or filesystem doesn't
follow symlinks.

## Option D — Tools without agent or skill support

For a plain chat tool (for example the ChatGPT or Claude web apps), use the
fallback CLI to generate a single Markdown session you paste in. See
[usage.md](usage.md#using-the-command-line-tool).

## Updating

- **skills CLI:** `npx skills update`.
- **Symlinked install:** `git pull` in your clone updates everything at once. Add
  a symlink only when a brand-new mode or skill appears.
- **Copied install:** re-copy the folders after a `git pull`.

## What each tool supports

An honest summary of the current picture. Tools differ in repository access,
whether they can write files, and whether they read custom agents, so this is not
a promise of identical behavior.

| Tool | Skills | Agent modes | Install |
| --- | --- | --- | --- |
| VS Code + Copilot | Yes (native) | Yes | Repo, symlink, or npx |
| GitHub Copilot CLI | Yes | Yes | Symlink or npx |
| Zed | Yes | No — use skills | npx or manual |
| Claude Code | Yes | No — use skills | npx or manual |
| Codex | Yes | No — use skills | npx or manual |
| Cursor | Yes | No — use skills | npx or manual |
| Plain ChatGPT / Claude chat | No | No | Fallback CLI (paste) |

Where a tool doesn't read the agent modes, the skills still carry the underlying
disciplines, and the fallback CLI can assemble a full mode into a pasteable
session.
